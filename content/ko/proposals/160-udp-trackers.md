## 상태

2025-06-24 검토에서 승인됨.
사양은 [UDP specification](/en/docs/spec/udp-bittorrent-announces/)에 있습니다.
zzzot 0.20.0-beta2에 구현되었습니다.
i2psnark API 0.9.67에 구현되었습니다.
다른 구현의 상태는 해당 문서를 확인하십시오.

## 개요

이 제안서는 I2P에서 UDP 트래커 구현을 위한 것입니다.

### 변경 기록

I2P에서 UDP 트래커에 대한 초기 제안이 2014년 5월 우리의 비트토렌트 사양 페이지 [/en/docs/applications/bittorrent/](/en/docs/applications/bittorrent/)에 게시되었습니다. 이는 우리의 공식 제안 프로세스에 앞서 있으며, 구현되지 않았습니다.
이 제안은 2022년 초에 작성되었으며 2014년 버전을 간소화합니다.

이 제안은 재사용 가능한 데이터그램에 의존하므로, 2023년 초에 Datagram2 제안 [/en/proposals/163-datagram2/](/en/proposals/163-datagram2/) 작업을 시작하면서 보류되었습니다.
그 제안은 2025년 4월에 승인되었습니다.

2023년 버전의 이 제안서는 두 가지 모드, "호환성"과 "빠름"을 명시했습니다.
추가 분석 결과 빠른 모드가 안전하지 않으며, 많은 수의 토렌트를 사용하는 클라이언트에게 비효율적일 것으로 나타났습니다.
추가로, BiglyBT는 호환성 모드를 선호한다고 표시했습니다.
이 모드는 표준 [BEP 15](http://www.bittorrent.org/beps/bep_0015.html)를 지원하는 모든 트래커나 클라이언트에 대해 쉽게 구현할 수 있습니다.

호환성 모드는 처음부터 클라이언트 크기에 구현하는 것이 더 복잡하지만, 2023년에 시작된 초안 코드가 있습니다.

따라서 현재 버전은 빠른 모드를 제거하고 "호환성" 용어를 제거하여 더욱 간단해졌습니다. 현재 버전은 새로운 Datagram2 형식으로 전환하고 UDP 발표 확장 프로토콜 [BEP 41](http://www.bittorrent.org/beps/bep_0041.html)에 대한 참조를 추가합니다.

또한, 이 프로토콜의 효율성을 확장하기 위해 연결 ID 수명 필드가 연결 응답에 추가되었습니다.

## 동기

일반 사용자 수와 특히 비트토렌트 사용자 수가 계속 증가함에 따라, 트래커가 과부화되지 않도록 트래커와 발표를 더욱 효율적으로 만들어야 합니다.

비트토렌트는 2008년 BEP 15 [BEP 15](http://www.bittorrent.org/beps/bep_0015.html)에서 UDP 트래커를 제안했으며, 지금은 대다수의 클리어넷 트래커가 UDP 전용입니다.

스트리밍 프로토콜에 비해 데이터그램의 대역폭 절감성을 계산하기는 어렵습니다. 재사용 가능한 요청은 스트리밍 SYN과 크기가 비슷하지만, 페이로드는 HTTP GET에 600바이트의 큰 URL 매개변수 문자열이 있어 약 500바이트가 적습니다.
원시 응답은 스트리밍 SYN ACK보다 훨씬 작아, 트래커의 발신 트래픽에 대해 상당한 절감을 제공합니다.

또한, 데이터그램은 스트리밍 연결보다 훨씬 적은 메모리 상태를 필요로 하므로 구현별 메모리 절감이 있어야 합니다.

[Prop169]에서 구상된 양자 이후 암호화 및 서명은 암호화 및 서명된 구조, 대상, 리스세트, 스트리밍 SYN 및 SYN ACK의 오버헤드를 상당히 증가시킬 것입니다. PQ 암호 채택 전에 이 오버헤드를 가능한 한 최소화하는 것이 중요합니다.

## 설계

이 제안서는 [/en/docs/spec/datagrams/](/en/docs/spec/datagrams/)에 정의된 재사용 가능한 데이터그램2, 재사용 가능한 데이터그램3 및 원시 데이터그램을 사용합니다.
Datagram2와 Datagram3는 제안서 163 [/en/proposals/163-datagram2/](/en/proposals/163-datagram2/)에서 정의된 새로운 재사용 가능한 데이터그램의 변형입니다.
Datagram2는 재사용 저항 및 오프라인 서명 지원을 추가합니다.
Datagram3는 이전 데이터그램 형식보다 작지만 인증은 없습니다.

### BEP 15

참고로, [BEP 15](http://www.bittorrent.org/beps/bep_0015.html)에 정의된 메시지 흐름은 다음과 같습니다:

```
클라이언트                         트래커
    연결 요청 --------------->
      <-------------- 연결 응답
    발표 요청 --------------->
      <-------------- 발표 응답
    발표 요청 --------------->
      <-------------- 발표 응답
```

연결 단계는 IP 주소 위조를 방지하기 위해 필요합니다.
트래커는 클라이언트가 후속 발표에서 사용하는 연결 ID를 반환합니다.
이 연결 ID는 기본적으로 클라이언트에서 1분 후 만료되고, 트래커에서는 2분 후 만료됩니다.

I2P는 기존 UDP 지원 클라이언트 코드 기반에서의 채택 용이성을 위해,
효율성을 위해, 그리고 아래 논의될 보안 이유로 BEP 15와 동일한 메시지 흐름을 사용할 것입니다:

```
클라이언트                         트래커
    연결 요청 ------------->       (재사용 가능한 데이터그램2)
      <-------------- 연결 응답   (원시)
    발표 요청 ------------->      (재사용 가능한 데이터그램3)
      <-------------- 발표 응답  (원시)
    발표 요청 ------------->      (재사용 가능한 데이터그램3)
      <-------------- 발표 응답  (원시)
             ...
```

이는 스트리밍(TCP) 발표보다 상당한 대역폭 절감을 제공할 가능성이 있습니다.
Datagram2는 스트리밍 SYN과 크기가 비슷하지만,
원시 응답은 스트리밍 SYN ACK보다 훨씬 작습니다.
후속 요청은 Datagram3를 사용하며, 후속 응답은 원시입니다.

발표 요청은 트래커가 연결 ID의 큰 매핑 테이블을 유지할 필요가 없도록 하기 위해 Datagram3입니다.
대신, 트래커는 연결 ID를 송신자 해시, 현재 타임스탬프(일정한 간격에 기초), 그리고 비밀값으로부터 암호학적으로 생성할 수 있습니다.
발표 요청이 수신되면, 트래커는 연결 ID를 검증하고 다음으로
Datagram3 송신자 해시를 송신 대상으로 사용합니다.

### 트래커/클라이언트 지원

통합 애플리케이션(예: i2psnark 및 ZzzOT Java 플러그인에서처럼 라우터와 클라이언트가 하나의 프로세스에 존재하는 경우)이나 I2CP 기반 애플리케이션(예: BiglyBT)은 스트리밍 및 데이터그램 트래픽을 별도로 구현하고 라우트하는 것이 비교적 간단해야 합니다.
ZzzOT와 i2psnark은 이 제안서를 구현할 첫 번째 트래커와 클라이언트가 될 것으로 예상됩니다.

비통합 트래커와 클라이언트는 아래에서 논의됩니다.

트래커
````````

현재 I2P 트래커 구현은 네 가지가 알려져 있습니다:

- zzzot, 통합 Java 라우터 플러그인, opentracker.dg2.i2p 및 기타 여러 곳에서 실행 중
- tracker2.postman.i2p, Java 라우터 및 HTTP 서버 터널 뒤에서 실행 중인 것으로 추정
- 오래된 C opentracker, UDP 지원이 주석 처리된 zzz에 의해 포팅됨
- 새로운 C opentracker, r4sas에 의해 포팅됨, opentracker.r4sas.i2p 및 다른 여러 곳에서 실행 중,
  i2pd 라우터 및 HTTP 서버 터널 뒤에서 실행 중인 것으로 추정

현재 발표 요청을 수신하기 위해 HTTP 서버 터널을 사용하는 외부 트래커 애플리케이션의 경우 구현은 꽤 어려울 수 있습니다.
데이터그램을 로컬 HTTP 요청/응답으로 변환하는 특수 터널을 개발할 수 있습니다.
또는 HTTP 요청과 데이터그램을 모두 처리하는 특수 터널이 설계되어 외부 프로세스로 데이터를 전달할 수 있습니다.
이 디자인 결정은 특정 라우터 및 트래커 구현에 크게 의존할 것이며, 이 제안의 범위를 벗어납니다.

클라이언트
`````````
SAM 기반 외부 토렌트 클라이언트(예: qbittorrent 및 기타 libtorrent 기반 클라이언트)는 i2pd에서 지원하지 않는 SAM v3.3 [/en/docs/api/samv3/](/en/docs/api/samv3/)이 필요할 것입니다.
이것은 또한 DHT 지원에 필요하며, 구현이 복잡하여 알려진 SAM 토렌트 클라이언트는 이를 구현하지 않았습니다.
이 제안서의 SAM 기반 구현은 곧 예상되지 않습니다.

### 연결 유효 기간

[BEP 15](http://www.bittorrent.org/beps/bep_0015.html)은 연결 ID가 클라이언트에서 1분, 트래커에서 2분 후 만료된다고 명시합니다.
이는 설정할 수 없습니다.
이는 잠재적인 효율성 이득을 제한하며, 클라이언트가 1분 내에 모든 발표를 마쳐야 하지 않는 한 가능하지 않습니다.
i2psnark은 현재 발표를 배치하지 않으며, 폭발적인 트래픽을 피하기 위해 발표를 분산시킵니다.
파워 사용자는 한 번에 수천 개의 토렌트를 실행하는 것으로 보고되었으며,
그렇게 많은 발표를 1분 내에 폭발시키는 것은 현실적이지 않습니다.

여기에서는 연결 응답을 확장하여 선택적 연결 수명 필드를 추가할 것을 제안합니다.
기본적으로, 지정되지 않으면 1분입니다. 그렇지 않으면, 초 단위로 클라이언트가 사용할 수 있으며, 트래커는
클라이언트 수명보다 1분 더 연결 ID를 유지합니다.

### BEP 15과의 호환성

이 디자인은 기존 클라이언트 및 트래커에 필요한 변경을 제한하기 위해 [BEP 15](http://www.bittorrent.org/beps/bep_0015.html)와 최대한 호환성을 유지합니다.

유일하게 필요한 변경은 발표 응답에서 피어 정보의 형식입니다.
연결 응답의 수명 필드 추가는 필수 사항은 아니지만,
위에서 설명한 대로 효율성을 위해 강력히 권장됩니다.

### 보안 분석

UDP 발표 프로토콜의 중요한 목표는 주소 스푸핑을 방지하는 것입니다.
클라이언트는 실제로 존재하고 실제 리스세트를 번들해야 합니다.
연결 응답을 받을 수 있는 인바운드 터널을 가져야 합니다.
이 터널은 0홉일 수 있으며 즉시 구축될 수 있지만, 이는 생성자를 노출시킬 것입니다.
이 프로토콜은 그 목표를 달성합니다.

## 사양

### 프로토콜 및 포트

재사용 가능한 데이터그램2는 I2CP 프로토콜 19를 사용합니다;
재사용 가능한 데이터그램3는 I2CP 프로토콜 20을 사용합니다;
원시 데이터그램은 I2CP 프로토콜 18을 사용합니다.
요청은 Datagram2 또는 Datagram3일 수 있습니다. 응답은 항상 원시입니다.
이전의 재사용 가능한 데이터그램("Datagram1") 형식은 요청 또는 응답에 사용되어서는 안 됩니다; 이러한 경우, 요청/응답 포트에서 수신되면 삭제되어야 합니다. Datagram1 프로토콜 17은 여전히 DHT 프로토콜에 사용되고 있음을 참고하십시오.

요청은 발표 URL에서 I2CP "to port"를 사용합니다; 아래를 참조하십시오.
요청 "from port"는 클라이언트가 선택하지만, 비제로여야 하며, DHT에서 사용하는 포트와 달라야 하여 응답을 쉽게 분류할 수 있어야 합니다.
트래커는 잘못된 포트로 수신된 요청을 거부해야 합니다.

응답은 요청에서 받은 "from port"로 I2CP "to port"를 사용합니다.
요청 "from port"는 요청의 "to port"입니다.

### 발표 URL

발표 URL 형식은 [BEP 15](http://www.bittorrent.org/beps/bep_0015.html)에서 지정되지 않았지만, 클리어넷에서처럼, UDP 발표 URL은 "udp://host:port/path" 형식입니다.
경로는 무시되며 비어 있을 수 있지만, 일반적으로 클리어넷에서는 "/announce"입니다.
:port 부분은 항상 존재해야 하지만, ":port" 부분이 생략되면, 클리어넷에서 일반적인 포트인 6969의 기본 I2CP 포트를 사용하십시오.
또한 &a=b&c=d와 같은 CGI 매개변수가 추가될 수 있으며, 이는 [BEP 41](http://www.bittorrent.org/beps/bep_0041.html)에서 설명한 대로 처리되고 발표 요청에 제공될 수 있습니다.
매개변수나 경로가 없으면, [BEP 41](http://www.bittorrent.org/beps/bep_0041.html)에서 암시하는 대로 후행 /도 생략될 수 있습니다.

```
오프셋  크기             이름            값
  0       32비트 정수  action          0 // 연결
  4       32비트 정수  transaction_id
  8       64비트 정수  connection_id
  16      16비트 정수  lifetime        선택적  // BEP 15에서의 변경 사항
```

응답은 요청 "from port"로 수신된 I2CP "to port"로 전송되어야 합니다.

lifetime 필드는 선택적이며, 초 단위로 연결 ID 클라이언트 수명을 나타냅니다.
기본값은 60이며, 지정된 경우 최소값은 60입니다.
최대값은 65535 또는 약 18시간입니다.
트래커는 클라이언트 수명보다 60초 더 연결 ID를 유지합니다.

- 6바이트 IPv4+포트 또는 18바이트 IPv6+포트 대신,
   32바이트 "압축 응답"의 배수로 SHA-256 바이너리 피어 해시를 반환합니다.
  TCP 압축 응답과 같이, 우리는 포트를 포함하지 않습니다.

응답은 요청 "from port"로 수신된 I2CP "to port"로 전송되어야 합니다.

I2P 데이터그램의 최대 크기는 약 64 KB로 매우 큽니다;
그러나 신뢰할 수 있는 전송을 위해, 4 KB 이상의 데이터그램은 피해야 합니다.
대역폭 효율성을 위해, 트래커는 아마도 최대 피어 수를 약 50으로 제한해야 하며,
이는 다양한 계층의 오버헤드 전에 약 1600바이트의 패킷에 해당하며,
단편화 후 두 개의 터널 메시지 페이로드 한도 내에 있어야 합니다.

BEP 15에서와 같이, 뒤따르는 피어 주소(IP/포트는 BEP 15에서, 해시는 여기) 수에 대한 계수는 포함되지 않습니다.
BEP 15에서는 포함되지 않았지만, 확장 데이터가 뒤따르는 경우 피어 정보가 완료되었음을 나타내기 위해 모든 0으로 된 끝 표시자를 정의할 수 있습니다.

미래의 확장이 가능하도록, 클라이언트는 32바이트 모든 0의 해시와 뒤따르는 데이터를 무시해야 합니다.
트래커는 모든 0의 해시로부터의 발표를 거부해야 하며,
이 해시는 이미 Java 라우터에 의해 금지되었습니다.

## 확장

확장 비트나 버전 필드는 포함되지 않았습니다.
클라이언트와 트래커는 패킷 크기를 특정 크기로 가정해서는 안 됩니다.
이 방식으로, 추가 필드를 추가해도 호환성이 깨지지 않습니다.
[BEP 41](http://www.bittorrent.org/beps/bep_0041.html)에서 정의된 확장 형식을 권장합니다.

연결 응답은 선택적 연결 ID 수명 필드를 추가하여 수정되었습니다.

블라인드된 목적지 지원이 필요한 경우, 우리는 발표 요청 끝에 블라인드된 35바이트 주소를 추가하거나, [BEP 41](http://www.bittorrent.org/beps/bep_0041.html) 형식을 사용하여 응답에서 블라인드된 해시를 요청할 수 있습니다(파라미터 TBD).
35바이트의 블라인드된 피어 주소 세트는 발표 응답 끝에, 모든 0으로 된 32바이트 해시 후에 추가될 수 있습니다.

## 구현 지침

비통합, 비-I2CP 클라이언트 및 트래커를 위한 과제에 대한 논의는 위 설계 섹션 참조하십시오.

### 클라이언트

특정 트래커 호스트 이름에 대해, 클라이언트는 HTTP URL보다 UDP URL을 선호해야 하며, 둘 모두에 발표해서는 안 됩니다.

기존 BEP 15 지원이 있는 클라이언트는 작은 수정만 필요할 것입니다.

클라이언트가 DHT 또는 기타 데이터그램 프로토콜을 지원하는 경우, 아마도 요청 "from port"로 다른 포트를 선택하여 응답이 그 포트로 돌아와 DHT 메시지와 혼동되지 않도록 해야 합니다.
클라이언트는 수신된 데이터그램3 패킷에 담긴 발신자 해시를 사용할 수 있기 때문에, 귀찮게 매핑 테이블을 유지할 필요가 없습니다.
트래커가 클라이언트에 전송할 때는 재사용 가능한 데이터그램2를 사용하지 않습니다.

기본 오프트래커 목록이 있는 클라이언트는 알려진 오프트래커가 UDP를 지원하는 후 UDP URL을 추가하도록 목록을 업데이트해야 합니다.

클라이언트는 요청의 재전송을 구현하거나 하지 않을 수 있습니다.
재전송을 구현하는 경우 최소 15초의 초기 타임아웃을 사용하고, 각 재전송에 대해 타임아웃을 두 배로 늘려야 합니다(지수 백오프).

클라이언트는 오류 응답을 받은 후 다시 요청을 시도하지 말아야 합니다.

### 트래커

기존 BEP 15 지원이 있는 트래커는 작은 수정만 필요할 것입니다.
이 제안서는 2014년 제안서와 달리 트래커가 동일한 포트에서 재사용 가능한 데이터그램2 및 3의 수신을 지원해야 합니다.

트래커 리소스 요구 사항을 최소화하기 위해
이 프로토콜은 트래커가 클라이언트 해시를 추후 검증을 위해 연결 ID로 매핑하는 것을
없애도록 설계되었습니다.
이는 발표 요청 패킷이 재사용 가능한 Datagram3 패킷이므로 송신자의 해시를 포함하기에 가능하다.

추천하는 구현 방법은 다음과 같다:

- 현재 시간을 수명으로 하는 현재 epoch을 정의하십시오.
  ``epoch = now / lifetime``.
- 8바이트 출력을 생성하는 암호화 해시 함수 ``H(secret, clienthash, epoch)``을 정의하십시오.
- 모든 연결에 사용할 랜덤 상수 비밀값을 생성하십시오.
- 연결 응답에 대해 ``connection_id = H(secret,  clienthash, epoch)``을 생성하십시오.
- 발표 요청에 대해 수신된 연결 ID가
  ``connection_id == H(secret, clienthash, epoch) || connection_id == H(secret, clienthash, epoch - 1)``인지를 검증합니다.

## 마이그레이션

기존 클라이언트는 UDP 발표 URL을 지원하지 않으며 무시합니다.

기존 트래커는 재사용 가능 데이터그램 또는 원시 데이터그램의 수신을 지원하지 않으며, 삭제됩니다.

이 제안서는 완전히 선택적입니다. 클라이언트나 트래커는 언제든지 구현할 필요가 없습니다.
